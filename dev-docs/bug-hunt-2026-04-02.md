# CLI Bug Hunt — 2026-04-02

Tested `logfwd` CLI as a first-time user following the README and docs.
Binary: `target/release/logfwd` built from master (`c0334c2`).

---

## Bug #1 — `--help` writes to stderr instead of stdout

**Reproduction:**
```sh
logfwd --help | grep endpoint   # returns nothing — help went to stderr
logfwd --help 1>/dev/null       # still shows help (it's on stderr)
```

**Expected:** `--help` writes to stdout so it can be piped/grepped.

**Actual:** `--help` writes to stderr. `--version` correctly writes to stdout. Inconsistent.

**File:** `crates/logfwd/src/main.rs` — `print_usage()` uses `eprintln!` throughout.

## Bug #2 — `--generate-json 0` prints `avg NaN bytes/line`

**Reproduction:**
```sh
logfwd --generate-json 0 /tmp/out.json
# Generating 0 JSON log lines to /tmp/out.json...
# done: 0.0 MB, avg NaN bytes/line
```

**Expected:** "0 bytes/line" or "N/A" — not `NaN`.

**Cause:** Division by zero — `total_bytes / num_lines` when `num_lines == 0`.

## Bug #3 — `--validate` success output goes to stderr instead of stdout

**Reproduction:**
```sh
logfwd --config config.yaml --validate 1>/dev/null   # still shows output
logfwd --config config.yaml --validate 2>/dev/null   # output disappears
```

**Expected:** Success diagnostics ("config ok: 1 pipeline(s)") go to stdout. Error messages go to stderr. The current behavior makes it impossible to check the exit code + capture the message reliably in a script.

## Bug #4 — Unknown top-level config keys are silently ignored

**Reproduction:**
```yaml
input:
  type: file
  path: /tmp/test.json
output:
  type: stdout
typo_field: something       # typo — should be 'server' or 'storage'
enrichement:                # typo — should be 'enrichment'
  labels:
    type: static
    fields:
      env: prod
```
```sh
logfwd --config above.yaml --validate   # exits 0, "config ok"
```

**Expected:** Validation error for unknown keys, or at minimum a warning.

**Impact:** Mistyped keys (e.g., `enrichement:` instead of `enrichment:`, `sever:` instead of `server:`) are silently swallowed. The user thinks their config is applied but it isn't.

## Bug #5 — `SELECT * FROM events` (wrong table name) passes `--validate`

**Reproduction:**
```yaml
transform: "SELECT * FROM events"
```
```sh
logfwd --config above.yaml --validate   # exits 0, "config ok"
```

**Expected:** Validation error: "table 'events' not found — the log table is always named 'logs'".

**Actual:** Passes validation, then at runtime every batch is dropped with:
```
pipeline: transform error (batch dropped): SQL execution error:
Error during planning: table 'datafusion.public.events' not found
```

The `datafusion.public.` prefix in the runtime error is also confusing to users who don't know DataFusion internals.

## Bug #6 — JSON output formatter strips `_str`/`_int`/`_float` from ALL column names, including user-defined `AS` aliases

**Reproduction:**
```sql
SELECT level_str, level_str AS myalias, duration_ms_int AS dur_int FROM logs LIMIT 1
```
JSON output:
```json
{"level":"INFO","myalias":"INFO","dur":1}
```

**Expected:**
- `level_str` → `level` (suffix strip, expected)
- `myalias` → `myalias` (no suffix, no strip, **expected**)
- `dur_int` → `dur_int` (user-defined alias, should not be mangled)

**Actual:** `dur_int` becomes `dur` — the suffix `_int` is stripped from the user's custom alias name.

**Impact:** A user who names a computed column `status_code_int` for readability would see `status_code` in the output with no explanation.

## Bug #7 — `SELECT *` silently enables `keep_raw`, injecting an undocumented `_raw` column

**Reproduction:**
```sql
SELECT * FROM logs
```
Output includes `_raw` with the full raw JSON line — even without any `keep_raw: true` setting in the config.

**Issues:**
1. `_raw` appears only for `SELECT *`, not for explicit column selects. `SELECT _raw FROM logs` fails with "No field named _raw".
2. Docs (`CONFIG_REFERENCE.md`) call this column `_raw_str` but it is actually `_raw`.
3. Users using `SELECT *` unknowingly double the data volume (each row carries its full raw JSON again).
4. Schema stability: cannot reference `_raw` by name in column-specific queries.

## Bug #8 — `COLUMN_NAMING.md` describes unreleased Phase-10 behavior; contradicts `README.md`

`COLUMN_NAMING.md` (added in commit `185c665`) says single-type fields use **bare names** (`level`, `status`, `duration_ms`). The current binary on master still emits old-style suffixed names (`level_str`, `duration_ms_int`, `service_str`).

Meanwhile `README.md` correctly describes the current behavior (always-suffixed names).

**Reproduction:**
```sql
SELECT level FROM logs WHERE level = 'ERROR'
-- Error: No field named level. Valid fields are logs.level_str, ...
```

A user following `COLUMN_NAMING.md` or `CONFIG_REFERENCE.md`'s column table would write SQL that fails at runtime.

**Affected docs:**
- `docs/COLUMN_NAMING.md` — describes bare names (future, not current)
- `docs/CONFIG_REFERENCE.md` SQL transform section — `_raw_str` / `_time_ns_int` / `_stream_str` / `_file_str` column names are all stale
- `README.md` SQL examples — use `level_str`, `status_int` (current) but `msg_str` which doesn't exist in generated data (`message_str` does)

## Bug #10 — Enrichment tables (`static`, `k8s_path`, `host_info`) are silently ignored

**Reproduction:**
```yaml
input:
  type: file
  path: /tmp/test.json
output:
  type: null

enrichment:
  labels:
    type: static
    fields:
      environment: production
```
```sh
logfwd --config above.yaml --validate   # exits 0, "config ok"
```

At runtime:
```
pipeline: transform error (batch dropped): SQL execution error:
Error during planning: table 'datafusion.public.labels' not found
```

**Root cause (two-part):**

1. The top-level `enrichment:` key is missing from `RawConfig`. In simple layout, `from_raw()` creates `PipelineConfig { enrichment: Vec::new(), ... }` — the enrichment block is silently dropped.

2. Even in advanced layout (if the schema accepted it), `EnrichmentConfig` only has one variant: `GeoDatabase`. The `static`, `k8s_path`, and `host_info` types are not in the enum — they would fail to deserialize.

**Impact:** The enrichment feature (heavily documented in README, CONFIG_REFERENCE, and examples) does not work for any of the documented types. Only `geo_database` is implemented.

## Bug #11 — `type: null` (YAML keyword collision) silently breaks advanced layout lists

**Reproduction:**
```yaml
pipelines:
  app:
    inputs:
      - type: file
        path: /tmp/test.json
    outputs:
      - type: null      # ← this is YAML null, not the string "null"
```
```
error: config YAML error: pipelines.app: data did not match any variant of
  untagged enum OneOrMany at line 4 column 5
```

**Cause:** In YAML, `null` is a reserved keyword for the null scalar value. `type: null` sets the `type` field to YAML null, not the string `"null"`. Serde fails to match any `OutputType` variant. The `OneOrMany` untagged enum masks the real error with a confusing message.

**Workaround:** Quote the value: `type: "null"`.

**Note:** `type: null` works in the simple layout (`output: {type: null}`) by accident, creating a user expectation that breaks in the advanced list context.

**Fix options:**
- Rename the output type from `null` to `devnull` or `discard`
- Or add a custom deserializer for `OutputType` that accepts YAML null → `Null` variant

## Bug #13 — `--validate` / `--check` flags must come AFTER `--config`, not before

**Reproduction:**
```sh
logfwd --validate --config myconfig.yaml
# error: unknown command: --validate
```

Users familiar with other CLI tools (e.g., `nginx -t -c config.nginx`) naturally put flags before the config path. The rigid positional parsing makes this confusing.

**Current valid form:**
```sh
logfwd --config myconfig.yaml --validate    # works
logfwd --validate --config myconfig.yaml    # fails
```

## Bug #14 — `grok()` UDF returns Struct instead of Utf8, crashing the pipeline

**Reproduction:**
```sql
SELECT grok('%{IP:client}', msg_str) AS parsed_str FROM logs
```
```
pipeline: transform error (batch dropped): Failed to collect results:
  Arrow error: Invalid argument error: column types must match schema types,
  expected Utf8 but found Struct([]) at column index 0
```

**Expected:** `grok(pattern, input) → utf8` (as documented in CONFIG_REFERENCE.md).

**Actual:** Returns a `Struct` type, causing an Arrow schema mismatch error whenever the result is collected. The UDF is completely non-functional.

## Bug #15 — Schema stability guarantee in COLUMN_NAMING.md is false

`COLUMN_NAMING.md` claims:
> Your SQL will never fail due to a missing column — it will simply return no matching rows for that column.

**Reproduction:**
```sql
SELECT level_str, nonexistent_str FROM logs LIMIT 1
-- Error: No field named nonexistent_str. Valid fields are logs.level_str.
```

The guarantee is not implemented. SQL that references columns absent from a batch fails at runtime.
