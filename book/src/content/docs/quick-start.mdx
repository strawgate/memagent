FastForward tails logs, filters them with SQL, and ships them to OTLP-compatible backends. This guide gets you from zero to a working pipeline in about 15 minutes.

:::note
The Cargo package is named `logfwd`, and the installed CLI binary is `ff`.
Docker images are currently published as `ghcr.io/strawgate/memagent`.
:::

## Install

<Tabs>
  <TabItem label="From Source" icon="seti:rust">
    Requires the Rust stable toolchain (1.89+).

    ```bash
    git clone https://github.com/strawgate/fastforward.git
    cd fastforward
    cargo build --release -p logfwd
    sudo cp target/release/ff /usr/local/bin/
    ```
  </TabItem>
  <TabItem label="Docker" icon="docker">
    ```bash
    docker run --rm \
      -v /var/log:/var/log:ro \
      -v $(pwd)/config.yaml:/etc/logfwd/config.yaml:ro \
      ghcr.io/strawgate/memagent:latest run --config /etc/logfwd/config.yaml
    ```

    Images are published to `ghcr.io/strawgate/memagent` for `linux/amd64` and `linux/arm64`. See [Docker deployment](/deployment/docker/) for compose files and volume configuration.
  </TabItem>
  <TabItem label="Kubernetes" icon="cloud">
    ```bash
    kubectl apply -f deploy/daemonset.yml
    ```

    See the [Kubernetes deployment guide](/deployment/kubernetes/) for resource sizing and collector integration.
  </TabItem>
</Tabs>

Verify it works:

```bash
ff --version
ff --help
```

## Stage 1: See log output

Generate synthetic JSON log lines and print them to your terminal.

```bash
ff generate-json 10000 logs.json
```

This creates 10,000 JSON log lines with fields like `level`, `message`, `status`, `duration_ms`, and `service`.

```yaml
# config.yaml
input:
  type: file
  path: logs.json
  format: json

output:
  type: stdout
  format: console
```

```bash
ff run --config config.yaml
```

You'll see colored output for every log line:

```
logfwd v0.x.x

  ✓  default
     in   file  logs.json
     out  stdout

ready · 1 pipeline (started in 38ms)

10:30:00.000Z  INFO   request handled GET /api/v1/users/10000  duration_ms=1 request_id=... service=myapp status=200
10:30:00.000Z  ERROR  request handled GET /health/10021  duration_ms=40 request_id=... service=myapp status=503
...
```

FastForward read the file, parsed each JSON line, and printed the results in a readable console format. All 10,000 lines should stream through quickly on a typical development machine.

:::note
File inputs keep tailing after they reach the current end of the file. For a finite one-shot workflow, pipe data into `ff send` instead.
:::

### Send piped data

For one-off command-line use, keep a destination-only config and pipe data into `ff`.
`ff send` reads stdin, drains the configured output or outputs, and exits.

```yaml
# destination.yaml
output:
  type: stdout
  format: console
```

```bash
cat logs.json | ff send --config destination.yaml --format json --service checkout
```

Bare `ff` also enters send mode when stdin is piped, using the normal config
search order. Set `LOGFWD_CONFIG` when you want an explicit destination config:

```bash
cat logs.json | LOGFWD_CONFIG=destination.yaml ff --format json --service checkout
```

## Stage 2: Filter with SQL

Add a SQL transform to keep only what matters. FastForward exposes each batch as a SQL table named `logs`.

```bash
ff generate-json 10000 logs.json    # regenerate — FastForward tracks file positions
```

```yaml
# config.yaml
input:
  type: file
  path: logs.json
  format: json

transform: |
  SELECT level, message, status, duration_ms
  FROM logs
  WHERE level = 'ERROR' AND duration_ms > 50

output:
  type: stdout
  format: console
```

```bash
ff run --config config.yaml
```

Now you see only errors with slow durations. Only the columns from the `SELECT` appear — everything else was filtered before reaching the output. In production, this means you only ship the logs you care about.

You can use standard SQL features such as `JOIN`, `GROUP BY`, `HAVING`, subqueries, and window functions. FastForward also includes helper functions such as `regexp_extract()`, `grok()`, `json()`, `json_int()`, `json_float()`, and `geo_lookup()`. See [SQL Transforms](/configuration/sql-transforms/) for the full reference.

:::caution
`ORDER BY` works within each batch but FastForward processes data in streaming batches, so ordering is per-batch, not global.
:::

## Stage 3: Ship to a collector

FastForward sends OTLP protobuf to an OpenTelemetry Collector, Grafana Alloy, or any OTLP-compatible backend.

Test locally with FastForward's built-in blackhole receiver:

```bash
ff blackhole &
```

```yaml
# config.yaml
input:
  type: file
  path: logs.json
  format: json

transform: |
  SELECT level, message, status, duration_ms
  FROM logs
  WHERE level IN ('ERROR', 'WARN')

output:
  type: otlp
  endpoint: http://127.0.0.1:4318/v1/logs
  compression: zstd
```

```bash
ff generate-json 10000 logs.json
ff run --config config.yaml
```

To ship to a real collector:

```yaml
output:
  type: otlp
  endpoint: https://otel-collector:4318/v1/logs
  compression: zstd
```

Works out of the box with [OpenTelemetry Collector](https://opentelemetry.io/docs/collector/), [Grafana Alloy](https://grafana.com/oss/alloy/), or any OTLP-compatible backend.

## Before you move on

Validate the config before deploying it anywhere more serious:

```bash
ff validate --config config.yaml
ff dry-run --config config.yaml
```

Both commands are read-only preflight checks: they parse the config, resolve paths and environment variables, and verify that inputs, outputs, and SQL can be built without starting ingestion. Use `validate` as the normal pre-deploy check; `dry-run` is the same safe build check with a more explicit command name.
