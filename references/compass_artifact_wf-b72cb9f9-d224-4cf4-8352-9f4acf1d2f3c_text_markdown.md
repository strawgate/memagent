# Proven kernel architectures in Rust: what works today

**The most effective pattern for separating a "proven core" in Rust combines three mechanisms: a dedicated `no_std` crate, `#![forbid(unsafe_code)]`, and trait-based abstraction at the boundary.** This is not theoretical — projects like rustls, s2n-quic, and CosmWasm ship production code with this architecture, and AWS runs Kani formal verification in CI on s2n-quic-core with every pull request. The pattern works, but the real-world evidence reveals sharp tradeoffs: the orphan rule is a show-stopping blocker for some projects, `no_std` eliminates HashMap, and no existing tool enforces per-crate dependency policies within a workspace.

## Six projects, three distinct strategies

Examining the six requested projects reveals three fundamentally different approaches to kernel isolation, each with different verification depths.

**Strategy 1 — Compiler-enforced purity (rustls).** The rustls TLS library represents the strongest architectural separation. The core `rustls` crate uses `#![forbid(unsafe_code)]` and `#![no_std]` unconditionally at the crate level. It contains the entire TLS protocol state machine, handshake logic, and record processing — all in safe, no-allocation-free Rust. Cryptography is abstracted behind a `CryptoProvider` struct with trait-based interfaces (`SecureRandom`, `SupportedKxGroup`, `KeyProvider`). The core crate has **zero** direct dependencies on any crypto implementation. Provider crates (`rustls-aws-lc-rs`, `rustls-ring`) implement these traits in separate workspace members. Since rustls 0.22, users must explicitly supply a `CryptoProvider` — there is no default. The enforcement mechanism is architectural: Cargo's dependency system makes it impossible for the core crate to accidentally depend on crypto implementations. Notably, **rustls does no formal verification at all**. Its security argument rests entirely on the compiler: 100% safe, `no_std` Rust cannot have memory safety bugs.

**Strategy 2 — Formal proofs in a core crate (s2n-quic).** AWS's s2n-quic takes a different approach, concentrating Kani formal verification proofs in its `s2n-quic-core` crate. The workspace splits into `s2n-quic-core` (protocol logic), `s2n-codec` (encoding primitives), `s2n-quic-platform` (IO), and provider crates. Core crates use conditional `no_std` via `#![cfg_attr(not(any(test, feature = "std")), no_std)]`. Rather than crate-level `forbid(unsafe_code)`, s2n-quic applies it per-module — the crypto, frame, interval_set, and transport error modules each carry `#![forbid(unsafe_code)]`. The breakthrough innovation is the **Bolero integration**: identical test harnesses run as concrete tests, fuzz tests, and Kani proofs via `#[cfg_attr(kani, kani::proof)]`. This means every property test is automatically a formal proof candidate. CI runs Kani on `s2n-quic-core` via a GitHub Action on every PR. The companion project s2n-tls (in C) goes further with SAW/Cryptol proofs of HMAC correctness, CBMC bounded model checking, and SideTrail timing analysis — the most mature continuous verification pipeline in production.

**Strategy 3 — Domain-specific language for verifiable rules (Wasmtime/Cranelift).** Cranelift's approach is architecturally unique: rather than verifying Rust code, they created **ISLE** (Instruction Selection and Lowering Expressions), a statically-typed term-rewriting DSL. ISLE rules define how CLIF IR instructions map to machine instructions. Because ISLE is declarative, its rules are amenable to SMT-based verification. The **Crocus** tool (ASPLOS 2024) proves full functional equivalence — that each rule translates an IR instruction to a machine code sequence with identical semantics. By 2023, the Wasmtime team had **formally verified all integer operation lowerings for the aarch64 backend**. Wasmtime also uses cargo-vet for supply chain security, with all dependencies audited by trusted maintainers and audit results shared bidirectionally with Firefox.

The remaining three projects fill in the picture. **CosmWasm** enforces determinism not through Rust's type system but through Wasm bytecode validation at upload time — rejecting floating-point operations (pre-1.5), whitelisting imports, and validating function bodies. Its `cosmwasm-core` crate is `#![no_std]`, but the real "kernel boundary" is the VM sandbox. **Ockam** uses `#![deny(unsafe_code)]` across most crates and isolates cryptographic keys behind a Vault trait (keys never leave the vault boundary), but despite the "build-trust" organization name, **no formal verification evidence exists** in its codebase. **Ferrocene** operates at a different level entirely — it qualifies the Rust compiler itself to ISO 26262 ASIL D and IEC 61508 SIL 4, with a growing subset of `core` separately certified (5,169 functions as of March 2026 at ASIL B/SIL 2).

## What `no_std` actually gives you — and what it costs

The `no_std` kernel pattern works on a three-tier model. The `core` crate provides primitives, `Option`, `Result`, `Iterator`, and (since Rust 1.81) `core::error::Error`. Adding `alloc` gives `Vec`, `String`, `Box`, `Rc`, `Arc`, `BTreeMap`, `BTreeSet`, and `VecDeque`. Full `std` adds `HashMap`, `HashSet`, filesystem, networking, threads, and IO.

The critical detail: **HashMap is not available in `alloc`**. It requires randomized hashing seeds for DoS protection, which demands OS entropy — something `alloc` deliberately avoids per RFC 2480. The deterministic alternative is `BTreeMap` (O(log n) vs O(1) average), or the `hashbrown` crate with a non-random hasher like `FxHash`.

The ergonomic costs are real but decreasing. Import paths change from `std::*` to `core::*` and `alloc::*`. Every dependency needs `default-features = false`. A custom panic handler is required for binaries. The biggest historical pain point — `Error` trait unavailable in `no_std` — was resolved when `core::error::Error` stabilized in Rust 1.81 (September 2024). The remaining gaps are floating-point math functions (need `libm`) and the compiler's inability to warn when a dependency silently pulls in `std` on hosted targets. The standard CI defense is compiling with `--target thumbv6m-none-eabi` to catch violations.

**Is `no_std` useful for provability?** Yes, but indirectly. Kani and other bounded model checkers cannot reason about external calls — OS syscalls, library FFI, network IO. By eliminating these, `no_std` constrains the state space to something BMC tools can actually explore. Amazon's verify-rust-std initiative targets `core` and `alloc` first precisely because they have fewer external dependencies. However, `no_std` alone is not a formal guarantee of purity. Unsafe code can still call FFI functions or execute inline assembly syscalls in a `no_std` crate. The true isolation requires `#![no_std]` + `#![forbid(unsafe_code)]` combined — this eliminates accidental IO dependencies *and* intentional escape hatches.

Kani explicitly supports `no_std` crates. The `kani_core` library is specifically designed for `no_std` and `no_core` environments. The test suite at `model-checking/kani` includes dedicated `no_std` verification examples. In practice, `extern crate kani;` is sufficient to use Kani in a `no_std` crate.

## No tool enforces per-crate dependency policies (yet)

The dependency restriction problem is the least solved aspect of the kernel architecture pattern. **cargo-deny does not support per-crate policies within a workspace** — this is tracked in issue #20, open since August 2019. It operates workspace-wide: you can ban `openssl` globally or use the `wrappers` feature to allow only one crate to depend on a dangerous dependency, but you cannot say "my-kernel-crate may only depend on these five crates."

**cargo-vet comes closest** with its `[policy]` table. Wasmtime's configuration demonstrates this: publishable crates like `cranelift` get `audit-as-crates-io = true` (all dependencies must be audited), while fuzz targets get `criteria = []` (no audit required). Custom criteria are possible — Google defines `ub-risk-0` through `ub-risk-4` for graduated unsafe code assessment. But cargo-vet controls audit requirements, not hard dependency bans.

The practical solution is layered defense. rustls uses **architectural separation** — separate crates with trait boundaries make it structurally impossible for the core to depend on crypto. Projects supplement this with `cargo tree` CI scripts that check dependency allowlists for specific crates:

```bash
cargo tree -p my-kernel-crate --prefix none -e normal \
  | awk '{print $1}' | sort -u | diff - allowed-deps.txt
```

For specific dependencies in a kernel crate: **memchr** is an excellent candidate (pure SIMD-optimized computation, no IO, widely audited by Mozilla and Google). **arrayvec** is ideal for strict kernels — fixed capacity, no allocation, ~1.5K SLoC audit surface. **smallvec** works but has had historical soundness issues (CVE-2021-25900). **bytes** is acceptable if the kernel already uses `alloc`, but its `Arc`-based reference counting adds complexity worth questioning.

## The orphan rule is the number one crate-splitting blocker

The Rust-for-Linux project provides the definitive case study on crate-splitting pain. When the team tried to extract `pin-init` as a separate crate (to share between kernel and userspace), they hit orphan rule violations: implementing a trait from the extracted crate on a type from `core` or `alloc` became illegal. The resulting issue (rust-lang/rust#136979) led to a 2024 H2 Rust language team project goal on relaxing the orphan rule. As project lead @ojeda stated: "Until we can relax the orphan rule, we're not able to split our crate. And in order to know how the orphan rule should be relaxed, we have to split our crate."

The newtype wrapper workaround exists but **propagates throughout the crate** and through all downstream consumers. Every place that touches the wrapped type needs conversion code. This is not merely inconvenient — it is architecturally corrosive.

Beyond the orphan rule, crate dependencies must be acyclic (unlike modules within a crate). The DFINITY/Internet Computer project (~350K lines of Rust) documented hitting this when splitting `interfaces` and `replicated_state` — they had to extract a shared `types` crate and use dependency inversion via traits. The standard resolution patterns are extracting shared type crates, trait-based inversion, or merging packages back together.

**Performance overhead at boundaries is a non-issue.** Monomorphization and LTO mean newtype wrappers and generic trait bounds compile away to nothing. The s2n-quic team confirmed that converting `Duration` to `u64` across a boundary produced identical machine code to hand-optimized code. The only real cost comes from opting into dynamic dispatch (`dyn Trait`) at boundaries.

The compilation tax is real but manageable. Each crate boundary adds Cargo.toml maintenance and prevents cross-crate inlining without LTO. The consensus from matklad (rust-analyzer, ~200K lines across 30+ crates) and others: use modules within a single crate for as long as possible, and only split into separate crates when you need procedural macros or want to enforce strict boundaries.

## Kani in production: s2n-quic and Firecracker lead the way

**s2n-quic is the canonical example of Kani proofs in a dedicated core crate.** Proofs are concentrated in `s2n-quic-core`, covering packet number round-trips, sliding windows, buffer deques, constant-time operations, and frame fitting. The Bolero framework unifies the harness: a single `#[test]` function annotated with `#[cfg_attr(kani, kani::proof)]` runs as a unit test, fuzz test, and formal proof. CI uses the `model-checking/kani-github-action@v0.17` GitHub Action targeting `quic/s2n-quic-core` with `--tests`. The Kissat SAT solver provides **200x speedups** over MiniSat on some harnesses (e.g., `gen_range_biased_test`: 1460s → 5.5s).

**AWS Firecracker** runs **27 Kani harnesses** across three verification suites in CI, completing in ~15 minutes. Kani found **five bugs in the rate limiter** and one in the VirtIO stack — including a subtle rounding error where integer division allowed guests to perform up to 0.01% more I/O than configured. This bug survived extensive conventional testing. Firecracker uses advanced Kani features: function stubbing (`#[kani::stub]`), contract verification (`#[kani::proof_for_contract]`), and verified stubs for compositional reasoning.

The **verify-rust-std** initiative represents the largest Kani deployment: verifying `core`, `alloc`, and `std` themselves. Of **9,078 unsafe functions** in `core`, 361 have been annotated with Kani function contracts. Challenges carry $5,000–$25,000 bounties. The **hifitime** crate used Kani to fix eight categories of bugs in a single PR — mostly at Duration boundary values (zero, maximum, minimum).

## Conclusion

The proven kernel architecture in Rust is not a single pattern but a **spectrum of verification depth**. At one end, rustls demonstrates that `#![forbid(unsafe_code)]` + `#![no_std]` + trait abstraction can provide strong isolation with zero formal verification — the compiler is the proof. At the other end, Cranelift's ISLE DSL enables SMT-based functional equivalence proofs of instruction lowering rules. In the middle, s2n-quic shows that Kani proofs can coexist with standard testing in a core crate, running in CI with manageable overhead.

Three facts should shape architectural decisions. First, **no existing tool enforces per-crate dependency restrictions within a workspace** — you must combine architectural separation, `no_std` targeting, and CI scripts. Second, **the orphan rule is a structural barrier** that the Rust language team is actively working to relax; plan type boundaries carefully or accept newtype wrappers. Third, **Kani works best on pure computational logic** — protocol state machines, codec operations, data structures — exactly the code that belongs in a `no_std` kernel crate. The pattern of concentrating formal proofs in a separated core crate is not just theoretically appealing; it is the architecture that s2n-quic, Firecracker, and the Rust standard library verification initiative have independently converged on.
